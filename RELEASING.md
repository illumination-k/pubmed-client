# Releasing

All publishable packages in this workspace share **one unified version**. A single
`v<semver>` tag releases everything, everywhere, in one run.

## Versioning policy

- **Single source of truth**: `[workspace.package] version` in the root `Cargo.toml`.
- Every other package is kept in lock-step with it by `scripts/sync-versions.sh`:
  - `[workspace.dependencies]` internal crate versions (`pubmed-parser`, `pubmed-formatter`, `pubmed-client`)
  - `pubmed-client-napi/package.json` — `version` **and** every `optionalDependencies.*` (per-platform binaries)
  - `pubmed-client-wasm/package.json` — `version` (the published `pkg/` is generated by wasm-pack from the crate `Cargo.toml`)
  - `pubmed-client-py/pyproject.toml` — `[project] version`
- CI enforces this on every PR via the **Version Consistency** job (`scripts/sync-versions.sh --check`).
  A drift makes CI red until `mise run sync-versions` is run.

There is intentionally **no** per-ecosystem versioning. Bumping the workspace version
bumps Rust crates, the npm packages, and the PyPI package together.

> **Version floor**: because all registries share one version, the new version must be
> **greater than the highest version already published to any registry** (crates.io / npm /
> PyPI). Publishing an existing version errors on crates.io and npm. The publish jobs are
> idempotent — they skip a registry where the version already exists — so a partially failed
> release can be safely re-run with the **same** tag; only a brand-new release needs a new version.

## Cutting a release

```bash
# 1. Bump every package to the new unified version + refresh Cargo.lock
mise run release 0.3.0

# 2. Review the diff, then commit
git commit -am "release: v0.3.0"

# 3. Tag and push — the tag is what triggers publishing
git tag v0.3.0
git push origin main --tags
```

The `v0.3.0` tag triggers [`.github/workflows/release.yml`](.github/workflows/release.yml),
which publishes in dependency order:

| Registry  | Packages                                                                            |
| --------- | ----------------------------------------------------------------------------------- |
| crates.io | `pubmed-parser` → `pubmed-formatter` → `pubmed-client` → `pubmed-cli`, `pubmed-mcp` |
| npm       | `pubmed-client` (NAPI, + platform binaries), `pubmed-client-wasm`                   |
| PyPI      | `pubmed-client-py`                                                                  |

It then creates the GitHub Release. Pre-release tags (`v0.3.0-rc.1`, `-alpha`, `-beta`)
are marked as prereleases on GitHub and still publish to every registry.

## Dry-run rehearsal

Trigger `release.yml` manually via **workflow_dispatch** (Actions tab) to run the entire
pipeline without publishing: crates use `cargo publish --dry-run`, npm uses
`npm publish --dry-run`, and Python publishes to **TestPyPI**.

## Workflow layout

- `release.yml` — the single entry point (tag `v*` or manual dry-run).
- `_publish-crate.yml` — reusable crates.io publisher (one call per crate).
- `_publish-napi.yml` — reusable NAPI build matrix + npm publish.
- `_publish-wasm.yml` — reusable wasm-pack build + npm publish.
- `_publish-python.yml` — reusable wheel build matrix + PyPI/TestPyPI publish.

CI workflows (`ci-napi.yml`, `ci-wasm.yml`, etc.) are build/test only and no longer publish.
